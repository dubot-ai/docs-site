## Keep the support stack. Add an action layer.

Dubot is designed to work alongside the tools a CX team already uses. The support platform remains the place where a request is understood. Dubot becomes relevant when resolving that request requires controlled work inside the product.

## The integration contract

A pilot integration should stay intentionally small. The exact mechanism depends on the support system and product environment, but the contract needs to answer five questions.

| Question | Minimum pilot answer |
| --- | --- |
| What starts a journey? | A named support intent or deliberate user choice |
| What context is passed? | Only the fields needed for the approved journey |
| How is the user identified? | An agreed identity and session model |
| Where does action occur? | A controlled product environment and surface |
| What returns to support? | A clear completed, stopped, or needs-attention result |

## A typical handoff

```text
Customer request
  -> support system identifies an eligible journey
  -> customer or agent chooses to continue
  -> Dubot receives the minimum required context
  -> approved journey opens in the product
  -> result is verified and reported
```

## What we do not need on day one

A useful pilot should not require a broad API program, a complete workflow catalog, or a production-wide rollout. We begin with one valuable request and the narrowest connection that can test it safely.
